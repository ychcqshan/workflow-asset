# EAM 项目开发经验与避坑指南 (Login 问题专项复盘)

## 📌 背景
在 Phase 2 阶段验证登录链路时，持续遇到了 `Request failed with status code 500` 和 `500用户名或密码错误` 的问题。本指南旨在复盘整个调试过程，并为后续新项目开发提供标准化的规避准则。

---

## 🔍 问题复盘与根本原因 (Root Cause Analysis)

纵观整个查错过程，导致登录失败实际上是**四个连续的陷阱**叠加造成的：

### 1. JDK 25 与 Lombok 的不兼容 (“隐藏杀手”)
- **现象**: 项目编译报错 `TypeTag :: UNKNOWN` 或前端收不到返回结果。
- **根因**: 项目本地环境中运行的是极新的 **JDK 25**。Lombok 依赖于 JDK 编译期的底层内部 API 修改 AST (抽象语法树)。高版本 JDK 封装了或移除了这些 API，导致 Lombok 的 `@Data` 完全失效。没有无参构造和 Get/Set，JSON 序列化全面崩溃。
- **解决**: 针对高版本 JDK 项目，**完全移除 Lombok**，改用显式的 Getter/Setter/Constructor。

### 2. Jackson 反序列化异常 (“实体类当参数用”)
- **现象**: 前端传入 `{"username":"admin", "password":"password"}`，后端返回 500，抛出 Jackson 解析异常。
- **根因**: 在移除了 Lombok 之后，原本直接使用 `SysUser` (数据库实体映射类) 来作为 `@RequestBody` 接收前端参数。由于缺少无参构造，或者包含了大量不相关的字段，导致 Spring Boot 的 `HttpMessageConverter` 无法正确反序列化 JSON。
- **解决**: 引入了专用的数据传输对象 **`LoginRequest` DTO** 来剥离与 DB 实体的强关联。

### 3. 命令行转义截断了 BCrypt 密文 (“哑巴吃黄连”)
- **现象**: 接口返回 “用户名或密码错误”。
- **根因**: 系统初始化的 `admin` 密码哈希是 `$2a$10$7JB...`。在使用 `docker exec` 或 PowerShell 管道导入初始化 SQL 时，因为字符串中含有大量的 `$` 符号，导致 Shell 自动进行 **变量插值 (Variable Interpolation)**。最终写进数据库里的密文丢失了一半段（从标准的 60 位被截断到了 36 位）。从而永远无法匹配成功。
- **解决**: 改为创建一个真实的临时 `.sql` 文件，通过重定向或挂载的方式灌入容器，规避 Shell 转义。

### 4. 开发僵尸进程导致代码未生效 (“薛定谔的修改”)
- **现象**: 明明修改了代码 (接入了 `BCryptPasswordEncoder`)，改了端口为 `9000`，测试依然走的是老逻辑，抛出新错误。
- **根因**: 终端通过 `mvn spring-boot:run` 开启的后台进程一直在后台挂着（持续跑了十几小时），没有随着 IDE 调试面板的停用而被彻底清理。后续改动的代码虽然在磁盘上更新了，但**运行在内存中的 JVM 进程还是老版本**。
- **解决**: 编写专门的 `stop.bat` 和 `start.bat` 根据系统的进程端口号进行 PID 级别的精确猎杀 (`taskkill`)，确保每次启动前释放端口，绝不留存僵尸进程。

## 🛠️ 后续项目开发标准规范 (Best Practices)

为了避免日后在其他项目遇到此类糟心问题，建议建立如下开发铁律：

### 规范 1：严格执行数据分层 (Entity vs DTO)
1. **绝不直接使用 Entity 类作为 Controller 层 `@RequestBody` 接收前端参数**。Entity 是映射数据库的，可能带有 JPA/MyBatis 的注解或逻辑字段，存在极大安全隐患（字段覆写注入）和序列化隐患。
2. 只要是入参，请老老实实写一个 `XXXRequest` DTO；只要是出参，写一个 `XXXResponse/VO`。

### 规范 2：环境脚手架隔离与自动化重置
1. 涉及安全相关的配置（密码、JWT Secret、Token 等）一旦带有特殊字符（`$`, `@`, `!`），严禁通过**内联命令 (Inline Commands)** 直接打进数据库或容器执行。
2. 测试用的数据初始化，务必以规范的 `init_xx.sql` 文件统一挂载至 Docker。
3. 把服务的启动、结束、清空容器封装成脚手架脚本 (比如此次编写的 `start.bat`和 `stop.bat`)，从第一天起就杜绝手工敲命令行开启后台服务造成的端口占用积压。

### 规范 3：框架依赖与环境基线锁定
1. 开设新项目时，若决定吃螃蟹使用最新版 Java（如 JDK 21+ / JDK 25），**请果断排雷 Lombok**，目前 Lombok 并不能完美兜底最新版的 Java 特性。与其天天猜为什么编译报错，不如回归 Java Record 或利用 IDE 快速生成 Get/Set 模版。
2. `pom.xml` 中的 Java 版本号必须与运行机器严格统一，明确开发基线。

### 规范 4：认证授权错误切勿静默
1. 当碰到 Auth/Login 报错时，永远**第一步去看日志的打印**，不要去猜。
2. 要求所有的认证拦截器 (Filter) 和 Login 接口，必须加上完整的 `log.debug()` 或者 `log.warn()`, 打印出到底卡在哪一步（是用户名查不到？还是密文不匹配？）。这在此次的调试转折点中起到了关键的指引作用。
