# 企业资产管理系统 (EAM) 系统设计方案

**文档版本**: v2.0 (优化版)
**拟定人**: 系统全栈架构师

---

## 1. 技术架构选型与扩展方案

为支撑企业级的高频并发（如季末集中盘点、批量发货领用）与多租户隔离，我们采用前后端分离的主流现代架构，并对中间件等进行扩展增强：

*   **业务后端 (Backend)**：Java 17 + Spring Boot 3 + Mybatis-Plus。全面采用响应式或高并发优化配置。
*   **持久层 (Database)**：MySQL 8.0（采用 InnoDB 引擎，保障事务一致性）。
*   **工作流引擎 (BPMN)**：Flowable 7.0（严格遵循 BPMN 2.0 规范，提供可视化能力与后端流程驱动）。
*   **前端界面 (Frontend)**：React 18 + Tailwind CSS + Vite (基于 Component-Driven Development)。使用 Lucide Icons 提供现代化图标。
*   **高速缓存与分布式锁 (Cache)**：**Redis 7.x**（新增扩展）。用于缓存系统的组织架构树、数据字典、用户 Session (JWT Token 黑名单) 以及防重复提交的分布式锁保障。
*   **分布式对象存储 (OSS)**：**MinIO**（新增扩展）。本地或私有云化部署的 OSS，用于存放资产图片、电子发票、电子签批凭证，彻底剥离数据库的 LOB 存储压力。
*   **监控与可观测性 (APM)**：Prometheus + Grafana（新增运维扩展视角），监控 JVM 状态、连接池及 API 延迟。

## 2. 核心架构设计原则

### 2.1 业务与流转的双轨制（解耦）

在实体关系中，我们将**物理资产流转（台账业务）**与**审批状态机（工作流引擎业务）**严格隔离，保证无论审批流配置如何修改，不会破坏历史台账的正确性。

*   **凭证关联**：业务表（`ams_request_bill`）记录申请详情及关联的 `proc_inst_id`。
*   **状态翻转**：资产业务表 `ams_asset` 的状态（如从“空闲”变为“在用”）不可人为直接修改，必须且只能由对应的流转**执行监听器 (ExecutionListener)** 或**任务监听器 (TaskListener)** 触发更新回调。

### 2.2 动态审批路由法则

*   **多级审批支持**：在 BPMN 设计器中，审批节点（如 UserTask）的 Assignee 必须配置为动态表达式，以支持复杂的层级审批（如“中心主任审批 -> 部门长审批”）。
*   **组织随动**：引擎在运行时到达该节点时，将自动回调后端的动态路由 Bean，通过解析发起人的 `dept_id` 自动在上岗组织树中灵活寻址对应的各级负责人。

### 2.3 API 接口契约规范

*   **RESTful 标准**：所有的 HTTP 接口遵循标准的 GET (查)、POST (增)、PUT (全量改)、PATCH (增量改)、DELETE (删) 语义。
*   **统一 JSON 响应封装**：
    ```json
    {
      "code": 200,             // 业务响应码 (不仅仅是 HTTP 200)
      "msg": "Success",        // 友好提示，或异常原因明文
      "data": { ... }          // 业务 Paylaod，分页场景会嵌套 records/total
    }
    ```

## 3. 增强版数据库模型 (ER 设想)

除了已有的核心台账表和审批关联单外，必须补充以下核心基石：

### 3.1 权限与组织架构 (RBAC 核心)
*   **`sys_user`**：用户基本信息与鉴权表。
*   **`sys_role` & `sys_menu`**：角色组定义表和前端动态路由/按钮指令权限表（`sys_role_menu`进行关联）。
*   **`sys_dept`**：组织树实体表，包含 `parent_id` 构筑树形结构，并扩展记录各种角色负责人信息（如部门长、中心主任等），支撑复杂的业务流程路由。

### 3.2 资产业务拓展
*   **`ams_asset`（主台账）**：包含字段 `asset_code` (唯一编码)、`category_id` (分类字典)、`buy_price` (原值)、`buy_date` (购入日)、`status` (状态: 0=空闲, 1=在用, 2=维修中, 9=已报废)、`current_user_id` (现持有人)。
*   **`ams_asset_record`（履历表）**：高频写入表。当资产发生任何流转（入、领、还、退、修、丢）时，异步或通过数据库触发器将当时的历史快照写入该表，形成时间轴。
*   **`ams_request_bill`（流程据点）**：承接所有的用户“申请”动作表单落地数据。

## 4. 系统安全性与鉴权设计

*   **JWT 令牌 (JSON Web Token)**：
    *   登录成功下发 accessToken 和 refreshToken。
    *   将登录用户信息序列化至 Token 中，后端服务通过网关的 `GatewayFilter` 进行解密和有效性校验，并将解析出的 `header: x-user-id` 传参给微服务节点。
*   **密码加密与数据安全**：
    *   数据库不准保存密码明文，全部通过 `BCryptPasswordEncoder` 单向散列加盐存储。
    *   配置项脱敏：使用类似 Jasypt 的库将 `application.yml` 中的 MySQL/Redis 密码进行加密保护。
*   **防止越权 (AOP 与 TaskService 拦截)**：
    *   对于资产数据的读写采用用户所属部门 `dept_id` 进行动态过滤组合拼装 SQL (运用 Mybatis-Plus 的拦截器插件)。
    *   对于任务的签收、办理操作执行前，通过引擎 API 查询任务候选人列表必须与当前 Token 解析操作人匹配，杜绝平行越权攻击。

## 5. 部署架构建议 (CI/CD DevOps)

*   建议所有的服务节点采用 **Docker 容器化部署**，统一发布到公司内部私仓。
*   运用 Kubernetes (K8s) 或更为轻量化的 Docker Compose 进行多节点编排与服务自愈重启编排。
*   Nginx / API Gateway 承担 HTTPS 证书卸载、跨域配置与反向代理负载均衡流量分发的工作。
